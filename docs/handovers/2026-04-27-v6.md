### 2. `add_team_code_member_and_wager`
- `kbt_teams` got a `team_code text` column + index + unique `(event_code, team_code)`
- New table `kbt_team_member`:
  ```
  id uuid pk default gen_random_uuid()
  team_id text not null            -- references kbt_teams.id
  event_code text not null
  team_code text                    -- denormalised for fast lookup
  player_name text not null
  player_code text not null         -- globally unique
  is_captain boolean not null default false
  created_at timestamptz not null default now()
  ```
  RLS on, anon SELECT/INSERT/UPDATE allowed (parity with `kbt_teams`).
- New table `kbt_wager`:
  ```
  id uuid pk default gen_random_uuid()
  event_code text not null
  team_name text not null
  round int not null
  question_number int not null
  wager_amount int not null default 0
  was_correct boolean
  created_at timestamptz not null default now()
  unique (event_code, team_name, round, question_number)
  ```
  RLS on, anon SELECT/INSERT/UPDATE allowed.
- Backfill: any existing `kbt_teams` row with a `team_captain` set but no member rows gets a captain `kbt_team_member` row created.

Plus a one-off seed: 12 quiz items inserted into `kbt_quiz` for `quiz_event_id = 5260` (TEST-001) across 2 rounds × 6 questions.

## How team / player / code mechanics work now

**Registration flow:**
- Captain opens player-app → fills team name, captain name, picks emoji → Register
- Client generates `team_code` (`T` + 6 chars from a 32-letter readability-friendly alphabet — no 0/O/1/I) and `player_code` (`P` + 6) on the device
- Inserts a row into `kbt_teams` with team_code + team_captain
- Inserts a `kbt_team_member` row marked `is_captain=true`
- Success screen shows the team_code prominently with three actions: Copy code, Copy join link, Share. Plus a QR (rendered via `api.qrserver.com`) that encodes the join URL.

**Joining flow:**
- Teammate opens `?code=EVENTCODE&team=TEAMCODE` (from QR scan or shared link) — auto-jumps to the join lookup
- Or opens player-app → "Have a team code? Join an existing team →" → enters code manually
- Looks up team via `kbt_teams.team_code = ?`, shows the team card, asks for the joiner's name
- Inserts a `kbt_team_member` row with `is_captain=false` and a fresh `player_code`
- Joined view shows team roster + the public teams-tonight list

**Captain editing:**
- On the captain's success screen there's an "edit" link next to the captain name
- Prompts for new name, PATCHes both `kbt_teams.team_captain` and the captain `kbt_team_member.player_name`

**Host-side visibility:**
- `loadTeamsFromSupabase` now pulls real `team_code` (carried through `kbt-data.js` → trialShaped)
- `displayTeams` lazy-loads `kbt_team_member` per team and shows player chips + their player_codes inside each team card
- Existing "Show QR" / "Copy" buttons work against the real codes
- Captain shown in the team header

## Gambler wager mechanic

In the live quiz host UI, when the current question's `q.type === 'Gambler'`:
- A bright magenta wager panel appears below the question
- One row per registered team: team name, wager amount input (0-20), ✓ Right / ✗ Wrong toggle
- "Lock in wagers — apply to round scores" button at the bottom
- On lock:
  - For each team that has a result selected: delta = wager × (+1 if right else -1)
  - Adds delta to `teamScores[team.id]['R'+round]`
  - Persists `kbt_wager` row (event_code, team_name, round, question_number, wager_amount, was_correct) — upserts via `merge-duplicates`
  - Persists the new round total via `submitRoundScore` → `trial_scores`
  - Re-renders score table

State held in `_wagerState[stateKey]` keyed by `r{round}q{qNum}` so navigating away and back preserves entered amounts/results until lock.

## Wrap pages (`wrap.html`)

Single-page Spotify-style recap, supports two modes:

- **Team mode** — `wrap.html?event=CODE&team=TEAMCODE`
- **Individual mode** — `wrap.html?event=CODE&player=PLAYERCODE` — resolves player → team_id → team, prepends an "Wrap for {name} — {role} of {team}" banner

Slides rendered top-to-bottom on a vertically scrollable mobile-shaped page (max 520px wide):

1. Hero — team emoji + name + venue + date (charcoal gradient)
2. Final rank — "1st of N teams" — gold gradient if first, magenta if last, blue otherwise — with copy varying by rank
3. Total points — huge number, with comparison to room average
4. Round-by-round bars (one per round, animated fill)
5. Peak round (gold)
6. Dip round (magenta)
7. The team — captain + all members as chips, captain chip in yellow
8. Tonight's top 3 — visual podium
9. Share — native share sheet on mobile, copy-link on desktop

Branding: KNOW + BRAINER lockup in the topbar, charcoal page bg, brand-color slide gradients, Fira Sans Condensed for headings, Outfit for body.

**For hosts to send wraps:** generate URLs in the form above. The team_code and player_code values come from `kbt_teams` and `kbt_team_member` respectively — readable in the host app.

## Outstanding gaps / next-up

- **VVDS Fifties font in Slides export** — Slides API can't load custom WOFF2; the export uses Open Sans as the closest substitute. If brand fidelity matters more, would need to upload a logo image and use it instead of text-based logo.
- **Per-question correctness tracking** — `kbt_live_answers` table exists but the host UI still only writes round totals. If you want richer wraps ("you got 24/30 right") this is the next layer.
- **Scoring is hardcoded to 3 rounds** — R1/R2/R3 only. If you ever run a 4-round or 5-round event, the score table needs to be parameterised.
- **`kbt_teams` RLS is still off** — known. Not a regression today, just flagged.
- **Stats card "Players" count** — currently equals team count (one player per team) until the host page implements a member count. Quick fix once members load — already loaded by `loadTeamMembersIntoCards`, just needs `updateStats` to sum players.
- **Two members can be marked captain** — no DB constraint preventing two `is_captain=true` rows for the same team_id. Recommend a partial unique index `(team_id) WHERE is_captain=true` if this becomes a problem.
