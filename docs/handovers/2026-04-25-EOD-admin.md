## Google Slides pipeline (`generateGoogleSlides()`)

The pipeline is fully implemented and deployed. It requires a user gesture to trigger (Google OAuth popup).

### Flow

1. User enters event code → click 🎯 Generate Google Slides
2. `getGToken()` → Google OAuth2 popup (user signs in, approves Slides + Drive scope)
3. Fetch event from `kbt_event` by event_code (or id, or description fuzzy match)
4. Fetch location from `kbt_loc`
5. Fetch quiz slots from `kbt_quiz` → get question IDs → fetch from `kbt_question`
6. Create blank Google Slides presentation via Slides API
7. Create blank Q+A slide stubs (2 slides per quiz slot = up to 64 slides)
8. Fetch source template presentations from `KBT_SOURCE_MAP` (cached per type)
9. Clone source elements into each stub via `buildCloneRequests()` + `batchUpdate` (chunked at 500 reqs)
10. Delete default blank slide
11. Fill placeholders globally + per-slide via `replaceAllText` (chunked at 500 reqs)
12. Show ✅ done + edit/present links

## Known gotchas

- `questions` and `kbt_question` are two separate tables. The slide gen uses `kbt_question`. The quiz generator section (`generateQuiz()`) uses `questions` via `db()`. Be careful not to mix them.
- `question_fun_fact` is used as `[question_supporttext]` in slides (not `question_question_supporttext`)
- The OAuth token is short-lived (~1hr). If slide gen fails with 401, just click generate again to re-auth.
- `buildCloneRequests()` clones background + all pageElements from source slides. If a source deck has very many elements, the chunk size of 500 may need reducing.
- The `dbFetch` function inside the slide gen closure is separate from the outer `db()` helper — both hit same Supabase.
